## 처리 흐름

1. 최초 결제를 요청한다.
2. 실패 응답을 내부 실패 코드로 변환한다.
3. 재시도 가능 여부와 다음 시각을 계산한다.
4. 가능하면 `RETRY_SCHEDULED`로 변경하고 작업을 예약한다.
5. 워커가 결제를 잠금 또는 원자적 갱신으로 선점한다.
6. 이미 성공했거나 취소된 결제면 종료한다.
7. 시도별 고유 멱등성 키로 PG에 요청한다.
8. 성공하면 결제를 `SUCCEEDED`로 변경한다.
9. 실패하면 다음 재시도를 예약하거나 `FAILED_FINAL`로 종료한다.

멱등성 키 예시:

```text
payment:{paymentId}:attempt:{attemptNumber}
```

- 주문 금액이나 통화가 변경되면 기존 결제를 재사용하지 않는다.
- 워커 장애에 대비해 오래된 `RETRYING` 상태를 복구한다.
- 로그에 카드번호, CVC 등 민감정보를 저장하지 않는다.
- 웹훅과 API 응답이 엇갈리면 PG의 최종 거래 상태를 기준으로 정합성을 맞춘다.

## 결제 실패 재시도 MVP

### 목표

일시적인 오류로 실패한 결제를 자동 재시도해 결제 성공률을 높이되, 중복 결제와 무한 재시도를 방지한다.

### 적용 범위

- 서버가 생성하는 단건 결제
- 네트워크 오류, PG 일시 장애 등 복구 가능한 실패
- 자동 재시도 최대 3회
- 운영자가 재시도 현황을 조회할 수 있는 로그
- 최종 실패 시 사용자 알림 연동 지점 제공

정기결제에도 같은 구조를 활용할 수 있지만, 구독 정지·유예기간·서비스 제한 정책은 별도 기능으로 분리한다.

## 핵심 정책

| 항목 | MVP 결정 |
|---|---|
| 최초 결제 | 재시도 횟수에 포함하지 않음 |
| 자동 재시도 | 최대 3회 |
| 간격 | 1분 → 5분 → 30분 |
| 재시도 대상 | 타임아웃, 네트워크 오류, PG 5xx, 일시적 처리 불가 |
| 즉시 종료 | 카드 분실·도난, 결제 거절, 유효기간 만료, 잘못된 요청, 인증 필요 |
| 중복 방지 | 하나의 결제 의도에 동일한 idempotency key 사용 |
| 동시 실행 | 동일 결제에 하나의 작업만 실행 |
| 최종 실패 | `FAILED_FINAL` 처리 후 알림 이벤트 발행 |
| 수동 재시도 | MVP 제외—새 결제 요청으로 처리 |

재시도 여부는 오류 메시지 문자열이 아니라 PG 오류 코드를 내부 분류로 변환해 판단한다. 알 수 없는 오류는 안전하게 재시도하지 않는 것을 기본값으로 둔다.

## 상태 모델

```text
PENDING
   │ 결제 요청
   ├── 성공 ───────────────> SUCCEEDED
   │
   └── 실패
        ├── 재시도 불가 ───> FAILED_FINAL
        └── 재시도 가능 ───> RETRY_SCHEDULED
                                  │
                                  └── RETRYING
                                       ├── 성공 ──> SUCCEEDED
                                       ├── 한도 미도달 ──> RETRY_SCHEDULED
                                       └── 한도 도달 ───> FAILED_FINAL
```

PG 응답이 불명확한 타임아웃은 바로 재결제하지 않는다. 먼저 PG 조회 API나 웹훅으로 기존 결제 성공 여부를 확인한 다음 재시도해야 한다.

## 처리 흐름

1. 결제 실패 응답을 내부 오류 분류로 변환한다.
2. 재시도 불가 오류면 즉시 최종 실패 처리한다.
3. 재시도 가능하면 다음 실행 시각을 저장하고 작업을 예약한다.
4. 작업 실행 시 결제 상태와 기존 PG 거래 결과를 다시 확인한다.
5. DB 잠금 또는 원자적 상태 변경으로 중복 작업을 차단한다.
6. 동일한 결제 의도와 멱등성 키로 PG를 호출한다.
7. 성공, 다음 재시도 예약, 최종 실패 중 하나로 상태를 확정한다.
8. 최종 결과 이벤트를 발행해 알림과 후속 처리가 구독하도록 한다.

작업 큐는 전달을 최소 한 번 보장한다고 가정하고, 소비자가 중복 실행돼도 안전하도록 구현한다.

## MVP 범위

### 1. 실패 분류

### 2. 재시도 정책

- 최초 실패 후 `1시간 → 24시간 → 72시간` 간격으로 최대 3회

- 결제가 성공·취소됐거나 주문이 만료되면 예약된 재시도 취소
- 운영자가 수동 재시도할 수 있지만 동일한 안전장치를 적용
- 재시도 횟수와 간격은 코드 상수 또는 설정값으로 관리

초기 버전에서는 오류별로 복잡한 스케줄을 다르게 두지 않는 편이 좋습니다.

### 3. 중복 결제 방지

- PG 요청에 `payment_id + attempt_no` 기반 idempotency key 전달
- 동시에 하나의 시도만 실행되도록 DB 잠금 또는 원자적 상태 변경 적용

